## Задание 3.2: Алгоритм создания пользователя на бэкенде

При получении HTTP POST-запроса на регистрацию (`/api/v1/users/register`), бэкенд-сервис должен выполнить пошаговую проверку. Главное правило: сервер не доверяет клиенту (фронтенду) и перепроверяет все данные заново.

### Пошаговый алгоритм:

**1. Проверка формата данных (Синтаксическая валидация)**
* Сервер проверяет, что в запросе пришли все обязательные поля: `firstName`, `lastName`, `username`, `password`, `captchaToken`.
* Сервер проверяет пароль: он должен быть не короче 8 символов, содержать цифры, заглавные/строчные буквы и спецсимвол (как указано в требованиях на макете).
* *Результат:* Если данные некорректны, сервер сразу прерывает работу и возвращает ошибку `400 Bad Request` с указанием, какое поле заполнено неверно.

**2. Проверка на бота (Валидация reCAPTCHA)**
* Сервер обращается к внешнему сервису-провайдеру антибот-защиты, чтобы проверить `captchaToken`. 
* *Результат:* Если токен невалидный (запрос прислал бот), возвращается ошибка `400 Bad Request`.

**3. Проверка уникальности логина (Бизнес-логика)**
* Сервер делает запрос к Базе Данных (БД), чтобы узнать, свободен ли присланный `username`.
* *Результат:* Если логин уже занят, сервер возвращает ошибку `409 Conflict` (текст: "Username is already exist!").

**4. Подготовка безопасности (Хеширование)**
* *Важное правило:* Пароли нельзя хранить в БД в открытом виде. 
* Сервер преобразует полученный пароль в специальный шифр (хеш) с помощью встроенных библиотек безопасности.

**5. Запись в Базу Данных**
* Сервер генерирует уникальный ID для нового пользователя.
* Сервер сохраняет в таблицу пользователей новую строку: ID, Имя, Фамилия, Логин и *хеш* пароля (не сам пароль!).
* *Результат:* Если во время сохранения пропадает связь с БД, возвращается ошибка `500 Internal Server Error`.

**6. Успешный ответ**
* Если запись успешно добавлена в базу, сервер возвращает фронтенду статус `201 Created` и ID нового пользователя.
